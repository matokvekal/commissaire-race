22# Role-Based Access Control (RBAC) Implementation Plan

## 📋 Overview

Implement comprehensive **EXTENSIBLE** RBAC system with permission matrix, token-based authentication, and view/edit modes for the commissaire race management application.

**🌟 KEY FEATURE**: The system is designed to be **fully extensible**:

- ✅ **Unlimited Custom Roles** - Create as many roles as needed
- ✅ **Dynamic Permissions** - Add new permissions for future features without code changes
- ✅ **Flexible Assignment** - Assign any combination of permissions to any role
- ✅ **Future-Proof** - New features (maps, analytics, etc.) automatically integrate

---

## 🚀 Quick Reference

### **Core Concepts at a Glance**

| Concept          | Description                                             | Example                                               |
| ---------------- | ------------------------------------------------------- | ----------------------------------------------------- |
| **Permission**   | A specific action a user can perform                    | `rider.add`, `maps.edit`, `results.publish`           |
| **Role**         | A collection of permissions                             | "Start Commissaire", "Course Marshal", "Photographer" |
| **Default Role** | Pre-configured role (cannot delete)                     | Admin, Start Commissaire, Finish Commissaire, Viewer  |
| **Custom Role**  | Admin-created role (unlimited)                          | "Media Staff", "Weather Monitor", "Race Announcer"    |
| **Token**        | Authentication credential containing role + permissions | Generated by admin for each user                      |
| **View Mode**    | Read-only mode (watch)                                  | All interactive elements disabled                     |
| **Edit Mode**    | Full interaction mode                                   | Based on user's permissions                           |

### **Permission Pattern: `feature.action`**

| Feature                  | Actions                                       | Examples                                          |
| ------------------------ | --------------------------------------------- | ------------------------------------------------- |
| **rider**                | add, edit, delete, view, status               | `rider.add`, `rider.edit`, `rider.view`           |
| **category**             | add, edit, delete, view, assign               | `category.add`, `category.assign`                 |
| **race**                 | create, edit, delete, start, finish, schedule | `race.start`, `race.finish`                       |
| **maps** _(future)_      | add, edit, delete, view, share                | `maps.add`, `maps.view`                           |
| **analytics** _(future)_ | view, export, advanced                        | `analytics.view`, `analytics.export`              |
| **[any new feature]**    | [any actions you define]                      | `weather.alert`, `media.upload`, `comm.broadcast` |

### **How It Works in 3 Steps**

1. **Admin creates role** → Select permissions from matrix → Save
2. **Admin generates token** → Give token to commissaire
3. **User logs in with token** → Gets access based on their role's permissions

**Adding New Feature in 3 Steps**:

1. **Developer registers permissions** → e.g., `maps.add`, `maps.edit`, `maps.view`
2. **Permission matrix updates automatically** → Admin sees new "Maps" permissions
3. **Admin assigns to roles** → Existing or new roles get map permissions

```typescript
// types/rbac.types.ts

export type Permission =
  // Riders
  | "rider.add"
  | "rider.edit"
  | "rider.delete"
  | "rider.view"
  | "rider.status"
  // Categories
  | "category.add"
  | "category.edit"
  | "category.delete"
  | "category.view"
  | "category.assign"
  // Standing
  | "standing.edit"
  | "standing.view"
  | "standing.lock"
  // Race
  | "race.create"
  | "race.edit"
  | "race.delete"
  | "race.start"
  | "race.finish"
  | "race.schedule"
  // Timing
  | "timing.record"
  | "timing.adjust"
  | "timing.view"
  // Results
  | "results.edit"
  | "results.publish"
  | "results.view";

export type RoleType =
  | "admin"
  | "start-commissaire"
  | "finish-commissaire"
  | "timer-commissaire"
  | "viewer";

export interface Role {
  id: string;
  name: RoleType;
  displayName: string;
  description: string;
  permissions: Permission[];
  isCustom: boolean;
  createdAt: Date;
  updatedAt: Date;
}

export interface User {
  id: string;
  name: string;
  email?: string;
  roleId: string;
  token: string;
  raceUuid?: string; // Optional: scope to specific race
  isActive: boolean;
  createdAt: Date;
  expiresAt?: Date;
}

export interface AuthToken {
  token: string;
  userId: string;
  roleId: string;
  permissions: Permission[];
  raceUuid?: string;
  issuedAt: Date;
  expiresAt?: Date;
}

export type ViewMode = "watch" | "edit";

export interface PagePermissions {
  canView: boolean;
  canEdit: boolean;
  mode: ViewMode;
  allowedActions: Permission[];
}
```

## � How to Add New Features with Permissions

### **Example: Adding Maps Feature**

When you want to add a new "Maps" feature to the app, follow these steps:

#### **Step 1: Define Permissions**

In `types/rbac.types.ts`, add new permissions to the Permission type:

```typescript
export type Permission =
  // ... existing permissions ...
  // Maps (NEW FEATURE)
  "maps.add" | "maps.edit" | "maps.delete" | "maps.view" | "maps.share";
```

#### **Step 2: Register Permissions**

In `services/auth/PermissionRegistry.ts`, register the new permissions:

```typescript
// Initialize when app starts
PermissionRegistry.registerPermissionGroup({
  category: "Maps",
  description: "Manage race course maps",
  isActive: true,
  permissions: [
    {
      key: "maps.add",
      label: "Add Maps",
      description: "Create new race course maps"
    },
    {
      key: "maps.edit",
      label: "Edit Maps",
      description: "Modify existing maps"
    },
    {
      key: "maps.delete",
      label: "Delete Maps",
      description: "Remove maps"
    },
    {
      key: "maps.view",
      label: "View Maps",
      description: "View race course maps"
    },
    {
      key: "maps.share",
      label: "Share Maps",
      description: "Share maps with other users"
    }
  ]
});
```

#### **Step 3: Update Admin Role**

Admin automatically gets all permissions, but you can update the default role definition:

```typescript
// In RoleService.ts - default roles
{
  name: 'admin',
  permissions: ['*'] // Wildcard = all permissions
}
```

#### **Step 4: Assign to Roles**

Now admin can:

1. Go to Permission Matrix
2. See new "Maps" column
3. Assign map permissions to existing roles (Start Commissaire, custom roles, etc.)

OR create a new "Map Manager" role:

1. Go to Roles Management
2. Click "Create New Role"
3. Name it "Map Manager"
4. Select all map permissions
5. Save

#### **Step 5: Use in Components**

In your Maps page component:

```typescript
import { useAuth } from '@/hooks/useAuth';
import PermissionGate from '@/components/auth/PermissionGate';

const MapsPage = () => {
  const { hasPermission } = useAuth();

  if (!hasPermission('maps.view')) {
    return <NoPermissionPage requiredPermission="maps.view" />;
  }

  return (
    <div>
      <h1>Race Maps</h1>

      {/* Only show Add button if user has permission */}
      <PermissionGate permission="maps.add">
        <button onClick={handleAddMap}>Add Map</button>
      </PermissionGate>

      {/* Map list with conditional edit/delete */}
      {maps.map(map => (
        <MapCard
          key={map.id}
          map={map}
          canEdit={hasPermission('maps.edit')}
          canDelete={hasPermission('maps.delete')}
        />
      ))}
    </div>
  );
};
```

#### **Step 6: Test**

1. Login as admin → should see all map features
2. Login as viewer → should only see maps (read-only)
3. Create "Map Manager" role → should see full map management
4. Generate token for Map Manager → test with that role

## �🔐 Security Considerations

1. **Token Security**
   - Use cryptographically secure token generation
   - Store tokens encrypted in IndexedDB
   - Implement token expiry
   - Support token revocation

2. **Permission Enforcement**
   - Check permissions on both frontend and backend (if applicable)
   - Never trust client-side permission checks alone
   - Validate permissions before any mutation

3. **Rate Limiting**
   - Limit failed login attempts
   - Throttle token generation

4. **Audit Logging**
   - Log all permission changes
   - Log token generation/revocation
   - Track user actions

## 🔄 Workflow: Adding a New Feature (Step-by-Step)

Let's say you want to add a **Weather Tracking** feature to your app:

### **Step 1: Plan Permissions**

Decide what actions users can take:

- `weather.add` - Add weather reports
- `weather.edit` - Edit weather conditions
- `weather.delete` - Remove weather reports
- `weather.view` - View weather dashboard
- `weather.alert` - Send weather alerts

### **Step 2: Update Type Definition**

```typescript
// types/rbac.types.ts
export type Permission =
  // ... existing permissions ...
  // Weather
  | "weather.add"
  | "weather.edit"
  | "weather.delete"
  | "weather.view"
  | "weather.alert";
```

### **Step 3: Register Permissions**

```typescript
// services/auth/PermissionRegistry.ts
PermissionRegistry.registerPermissionGroup({
  category: "Weather",
  description: "Track and alert on weather conditions",
  isActive: true,
  permissions: [
    {
      key: "weather.add",
      label: "Add Reports",
      description: "Log weather conditions"
    },
    {
      key: "weather.edit",
      label: "Edit Reports",
      description: "Update weather data"
    },
    {
      key: "weather.delete",
      label: "Delete Reports",
      description: "Remove weather logs"
    },
    {
      key: "weather.view",
      label: "View Dashboard",
      description: "See weather dashboard"
    },
    {
      key: "weather.alert",
      label: "Send Alerts",
      description: "Alert commissaires of dangerous weather"
    }
  ]
});
```

### **Step 4: Build the Feature**

Create your weather component with permission checks:

```typescript
const WeatherPage = () => {
  const { hasPermission } = useAuth();

  if (!hasPermission('weather.view')) {
    return <NoPermission />;
  }

  return (
    <div>
      <h1>Weather Dashboard</h1>

      <PermissionGate permission="weather.add">
        <button onClick={addReport}>Log Weather</button>
      </PermissionGate>

      <PermissionGate permission="weather.alert">
        <button onClick={sendAlert}>Send Alert</button>
      </PermissionGate>
    </div>
  );
};
```

### **Step 5: Admin Assigns Permissions**

- Admin opens Permission Matrix
- Sees new "Weather" column automatically
- Assigns weather permissions to roles:
  - **Admin**: All weather permissions
  - **Start Commissaire**: `weather.view`, `weather.add`
  - **Race Director**: All weather permissions
  - **Course Marshal**: `weather.view`, `weather.alert`
  - **Viewer**: `weather.view` only

### **Step 6: Create Specialized Role (Optional)**

Admin creates "Weather Monitor" role:

- All weather permissions
- `comm.broadcast` (to alert everyone)
- `rider.view` (see riders on course)
- No race control permissions

### **Step 7: Done!**

- Feature works immediately
- All roles updated
- Tokens include new permissions
- No system downtime
- No database migrations
